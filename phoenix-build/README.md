# phoenix-build — a Phoenix 1.8 + LiveView app built hermetically by JOBS

This example builds a real **Phoenix 1.8** application (app name `fireside`)
fully **offline** (`net=none`) in the JOBS hermetic sandbox, with **no `/nix`
and no glibc**. It exercises the full stack a modern Phoenix app needs:

- **Offline hex.** The new **`mixplugin`** turns `mix.lock` (33 hex entries)
  into one content-addressed `.tar` import per package via the new
  **`hexpm`** fetcher, pinned by the lock's own outer checksum — staged
  straight into mix's package cache, so `HEX_OFFLINE=1 mix deps.get`
  re-verifies every package against that checksum with zero network access.
- **Compiling C in a NIF.** `exqlite` bundles its own SQLite and compiles the
  C NIF *from source inside the sandbox* with **`zig cc`** targeting musl.
- **A real DB roundtrip.** The build runs an in-sandbox Ecto migration and a
  DB roundtrip; the build's success is gated on that roundtrip succeeding,
  with `PHOENIX-OK` printed as the marker.
- **Offline JS/CSS.** Phoenix 1.8's asset pipeline — **Tailwind v4 + daisyUI +
  esbuild** — runs fully offline from a pre-fetched npm mirror via a **musl
  Node** (**`nodeplugin`** over `assets/package-lock.json`; the hex `esbuild`/
  `tailwind` wrappers exec the npm-installed CLIs via `ESBUILD_PATH`/
  `TAILWIND_PATH` instead of downloading their own binaries).
- **A deployable release.** Output is a self-contained `mix release` (bundled
  ERTS) plus the already-migrated SQLite DB and a `bin/serve` entrypoint —
  runnable with `jobs run`, deployable with `jobs image`.

## The offline-hex trust story

`mix.lock`'s outer checksum (the second hash in each lock entry) **is the
sha256 of the package tarball on `repo.hex.pm`** — there's no separate trust
gap to bridge. `mixplugin` uses that checksum directly as the content address
for the `hexpm` import, and hex re-verifies it again itself at `deps.get`
time, so the build is checked twice against the same value: once by JOBS's
content-addressed fetch, once by hex's own offline verification.

Git dependencies are a **hard error** in `mixplugin` by design — a lockfile
entry naming a git source has no stable checksum to key an import on. Stock
Phoenix 1.8.9 pulls in two git deps this way (`heroicons` and daisyUI's
upstream), so both are stripped from the dependency graph: `core_components.ex`
renders a trivial placeholder glyph instead of `Heroicons`, and daisyUI ships
as an ordinary npm package (offline mirror) rather than a git-vendored hex
dep.

## `hex-cache.ets` (committed)

Hex 2.x needs a registry snapshot at `$HEX_HOME/cache.ets` before it can
resolve/verify packages offline — this repo commits that snapshot as
`hex-cache.ets` at the app root, generated by the **same hex version the
build pins** (`2.2.1`). Regenerate it whenever `mix.lock` changes or the hex
pin moves, on any networked machine:

```bash
export HEX_HOME=$(mktemp -d)          # fresh, don't reuse an existing one
mix archive.install <path-to-hex-2.2.1.ez> --force
mix deps.get
cp "$HEX_HOME/cache.ets" hex-cache.ets
```

## `rebar3`

Some hex packages (e.g. `telemetry`) are built with rebar3, not `mix`.
`rebar3` is pinned from its **byte-stable GitHub release asset** —
`repo.hex.pm/installs/.../rebar3` is demonstrably *not* byte-stable across
fetches, so it can't be used as a content-addressed import.

## Build it & run it

```bash
scripts/dev-setup.sh                               # one-time: local env (embedded store) + signing key
jb run --source ../examples/phoenix-build          # build offline, then serve on :4000
# or: jb develop --source ../examples/phoenix-build  # drop into the build sandbox
```

```
$ curl localhost:4000
<!-- ... -->
<h1 ...>Fireside guestbook</h1>
<!-- ... -->
```

The build itself prints `PHOENIX-OK exqlite NIF + Ecto roundtrip verified`
once the in-sandbox migration + DB roundtrip gate passes; `curl` against the
running release then proves the rest at runtime — the OTP release boots,
Ecto/SQLite serve real data, and the offline-compiled Tailwind/esbuild
assets are what render the page.

## Declared build caches

Three persistent caches, atime-pruned, gated to their own `build-pinned:F`:

- **`/zig-cache`** — zig's C compile cache for the `exqlite` NIF.
- **`/mix-build-cache`** — mix's `_build` (via `MIX_BUILD_ROOT`), keyed
  `elixir1.18-otp27`.
- **`/mix-deps-cache`** — the extracted `deps/` tree. Keeping dependency
  source mtimes stable across builds is what lets mix skip recompiling the
  whole dependency closure on a warm rebuild.

Locally, a warm rebuild (editing only app code) took **1m20s** vs **2m14s**
cold — the dependency closure fetch/extract and the `exqlite` NIF compile are
both skipped.

## Notes / pins

- Alpine **v3.22** apks: `erlang27` (OTP 27), `elixir` 1.18.3, musl userland
  (the literal `erlang27`/`erlang27-dev` package names — plain `erlang`/
  `erlang-dev` are provides-only virtual names in v3.22).
- zig **0.16.0** (the in-sandbox C compiler for the `exqlite` NIF).
- musl **Node v22** (assets pipeline: Tailwind v4, daisyUI, esbuild, all via
  the offline npm mirror).
- Platform-parameterized: builds on **amd64 and arm64**.
